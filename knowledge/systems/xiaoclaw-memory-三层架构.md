# xiaoclaw-memory 三层记忆架构

## 概述

xiaoclaw-memory 是为 OpenClaw AI 助手设计的**分层记忆系统**，模仿人类记忆的工作方式：

- **L0**：轻量索引（快速检索）
- **L1**：分类记忆（长期存储）
- **L2**：每日日志（原始记录）

核心理念：**Text > Brain** — 写下来，而不是记在「心里」。

## 各层详解

### L0 层 — 索引层

**文件**：`MEMORY.md`

**作用**：
- 快速了解记忆系统全貌
- 按需加载对应的 L1 文件
- 减少每次会话的 token 消耗

**内容结构**：

```markdown
# MEMORY.md - 长期记忆索引

## 📊 记忆统计

| 类型 | 标签 | 数量 | 文件 | 最后更新 |
|------|------|------|------|----------|
| Profile | `[P]` | 4 | `memory/profile.md` | 2026-03-01 |
| Event | `[E]` | 2 | `memory/events.md` | 2026-03-02 |
| Knowledge | `[K]` | 6 | `memory/knowledge.md` | 2026-03-02 |
| Behavior | `[B]` | 8 | `memory/behaviors.md` | 2026-03-02 |
| Skill | `[S]` | 9 | `memory/skills.md` | 2026-03-02 |

## 📂 文件摘要

### profile.md — 用户画像 [P]
Kevin，技术从业者，关注 AI/GitHub/Twitter...

### knowledge.md — 技术知识 [K]
LanceDB 架构、xiaoclaw-memory 设计、GitHub CLI...
```

**加载规则**：
- 每次新会话**必读**
- 根据任务按需加载对应的 L1 文件

---

### L1 层 — 分类记忆层

**文件**：`memory/*.md`

**分类**：

| 类型 | 标签 | 文件 | 内容 |
|------|------|------|------|
| **Profile** | `[P]` | `profile.md` | 用户画像、偏好、背景 |
| **Event** | `[E]` | `events.md` | 重要事件、里程碑 |
| **Knowledge** | `[K]` | `knowledge.md` | 技术知识、客观信息 |
| **Behavior** | `[B]` | `behaviors.md` | 行为模式、规则、习惯 |
| **Skill** | `[S]` | `skills.md` | 技能经验、方法论 |

**记录格式**：

```markdown
### [类型] 标题

- **字段1**: 值1
- **字段2**: 值2
- **关联**: → 其他文件#锚点
- **日期**: YYYY-MM-DD
```

**加载规则**：
- 根据任务按需加载
- 不全量加载（避免 token 浪费）

---

### L2 层 — 每日日志层

**文件**：`memory/YYYY-MM-DD.md`

**作用**：
- 记录当天的原始对话、决策、发现
- 短期存储，避免「记忆碎片」
- 定期蒸馏到 L1

**记录格式**：

```markdown
# 2026-03-02 日志

## [E] 修复定时任务投递问题
- **问题**: xxx
- **解决**: xxx

## [S] 新增 OpenFang Hands 设计原则
- **来源**: xxx
- **原则**: xxx

## [K] cron announce 投递问题
- **现象**: xxx
- **临时方案**: xxx
```

**蒸馏规则**（定期执行）：
1. 读取最近 3 天的 L2 日志
2. 按类型标签分拣到对应的 L1 文件
3. 合并重复条目（保留最新版本）
4. 添加交叉引用
5. 更新 L0 索引

## 类型标签详解

### [P] Profile — 用户画像

记录用户的：
- 基本信息（姓名、职业、时区）
- 兴趣关注（技术方向、信息来源）
- 偏好设置（沟通风格、输出格式）

**示例**：
```markdown
### [P] 基本信息
- **姓名**: Kevin
- **职业**: 技术从业者
- **时区**: Asia/Shanghai (GMT+8)
```

---

### [E] Event — 重要事件

记录：
- 里程碑事件
- 重要决策
- 系统变更

**示例**：
```markdown
### [E] 部署 xiaoclaw-memory 系统
- **日期**: 2026-03-01
- **内容**: 完成三层记忆架构的设计与部署
- **影响**: AI 助手拥有了长期记忆能力
```

---

### [K] Knowledge — 技术知识

记录：
- 技术架构
- 工具使用方法
- 客观知识

**示例**：
```markdown
### [K] LanceDB 架构
- **类型**: 向量数据库
- **特点**: 嵌入式、无服务器、支持向量搜索
- **用途**: AI 记忆系统底层存储
```

---

### [B] Behavior — 行为模式

记录：
- AI 的行为规则
- 工作流程
- 用户偏好

**示例**：
```markdown
### [B] 记忆规则
- 只记录用户明确指令的内容
- 不自动记录所有对话
- **更新记忆时回复格式**: 用表格列出更新的文件
```

---

### [S] Skill — 技能经验

记录：
- 技术选型决策
- 开发方法论
- 踩坑经验

**示例**：
```markdown
### [S] OpenFang Hands 设计原则
- **来源**: [OpenFang](https://github.com/RightNow-AI/openfang)
- **可借鉴的设计**:
  1. Settings 可配置化
  2. System Prompt 分阶段
  3. 配置与逻辑分离
```

## 工作流程

### 写入流程

```
新信息 → 写入 L2 日志（标注类型标签）→ 判断是否有长期价值
                                          ↓ 有
                                    提炼到 L1 文件
                                          ↓
                                    更新 L0 索引
```

### 读取流程

```
新会话启动 → 读取 L0 索引
                  ↓
            根据任务判断需要哪些 L1
                  ↓
            按需加载对应的 L1 文件
```

### 蒸馏流程（定期执行）

```
读取最近 3 天 L2 日志
        ↓
按类型标签分拣
        ↓
写入对应 L1 文件
        ↓
更新 L0 索引统计
```

## 与其他系统的关系

### 与 Capability Evolver

| xiaoclaw-memory | Capability Evolver |
|-----------------|-------------------|
| `memory/*.md` | 只读扫描 |
| 独立存储 | `memory/evolution/` |
| 目的：记录用户信息 | 目的：自我进化 |

两者**互补不冲突**。

### 与 Obsidian 协作仓库

| xiaoclaw-memory | Obsidian 仓库 |
|-----------------|---------------|
| AI 运行时记忆 | 人机协作笔记 |
| `memory/*.md` | `obsidian-vault/` |
| 自动读写 | 需确认后同步 |

**同步规则**：写入 Obsidian 前需 Kevin 确认。

## 设计原则

1. **Text > Brain** — 写下来，不靠「记忆」
2. **分层加载** — L0 必读，L1 按需，L2 蒸馏
3. **类型标签** — `[P/E/K/B/S]` 便于分拣
4. **交叉引用** — `→ 文件名#锚点` 建立关联
5. **定期维护** — 蒸馏、清理、更新索引
